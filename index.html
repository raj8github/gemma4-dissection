03 — Embeddings

The Embedding Table IS the Vector Database

In zero2ai you learned every token gets a "rich profile card" — a vector encoding its meaning. MicroGPT had wte[27×16]. Gemma 4 is the exact same lookup: 262,144 tokens × 5,376 numbers each.

When people ask "where's the vector database?" — the embedding table IS it. 262K × 5376 floats (~2.8 GB) stores every token's meaning. Similar meanings end up geometrically close. No separate vector DB — it's baked into the weights. Token ID → row index → 5376-dim vector.

Embedding lookup — animating continuously
# embed_tokens: [262144, 5376] — 2.8 GB # tie_word_embeddings = true: # LM head reuses this matrix transposed # → saves another 2.8 GB of storage token_id = 4821 # "▁model" vec = embed_tokens[4821] # → shape [5376] — the token's meaning

Why 5376 Dimensions?

More dimensions = more concept axes. Each can represent a different meaning direction.

04 — Decoder Layers

60 Layers, Two Alternating Types

MicroGPT used 2 transformer blocks. Gemma 4 is the same structure — attention then MLP — 60 times. The key difference: not all 60 are identical. They alternate between two attention modes.

From config.json: 5 sliding-window layers, then 1 full-attention layer, repeated 10 times. Every 6th layer sees the full 256K sequence. Layer #59 is always full-attention.

All 60 layers — click any to inspect its weight matrices
■ Sliding window   ■ Full global (every 6th)
Select a layer to inspect all its weight matrices.

Why 5:1 ratio? Full attention across 256K tokens is O(n²) — 65 billion ops per layer. 50 sliding layers (O(n × 1024)) handle local syntax cheaply. 10 global layers (every 6th) integrate long-range context. The model gets both speeds for the cost of one.

Every Layer Has the Same 14 Weight Tensors

# Per-layer weights stored in safetensors files: layers.N.input_layernorm.weight # [5376] normalise before attention layers.N.self_attn.q_proj.weight # [8192, 5376] query: what am I looking for? layers.N.self_attn.k_proj.weight # [4096, 5376] key: what do I contain? layers.N.self_attn.v_proj.weight # [4096, 5376] value: what to extract? layers.N.self_attn.q_norm.weight # [256] NEW: RMSNorm on Q vectors layers.N.self_attn.k_norm.weight # [256] NEW: RMSNorm on K vectors layers.N.self_attn.o_proj.weight # [5376, 8192] combine all heads → hidden layers.N.post_attention_layernorm.weight # [5376] layers.N.pre_feedforward_layernorm.weight # [5376] layers.N.mlp.gate_proj.weight # [21504, 5376] MLP gate (GELU) layers.N.mlp.up_proj.weight # [21504, 5376] MLP content layers.N.mlp.down_proj.weight # [5376, 21504] compress back to hidden layers.N.post_feedforward_layernorm.weight # [5376] layers.N.layer_scalar # [1] NEW: per-layer volume dial
05 — Attention

Sliding Window vs Full Attention

In zero2ai you saw attention as "every word looking at every other word." In microGPT you saw the Q·K·V math. Gemma 4 uses the same Q·K·V — but limits the look-back range in 50 of 60 layers.

Attention pattern — toggle between modes
Each token attends only within its 1024-token window — fast and memory-efficient

Sliding Layer (50 of 60)

# 32 Q heads, 16 KV heads (GQA) # head_dim: 256, window: 1024 tokens Q = x @ q_proj.T # [seq, 8192] K = x @ k_proj.T # [seq, 4096] Q = rms_norm(Q, q_norm) # stabilise Q K = rms_norm(K, k_norm) # stabilise K scores = Q @ K.T / sqrt(256) # mask: only last 1024 positions out = softmax(scores) @ V out = out @ o_proj.T # [seq, 5376]

Global Layer (every 6th)

# 32 Q heads, only 4 KV heads # head_dim: 512, no window mask # attention_k_eq_v = true Q = x @ q_proj.T # [seq, 16384] K = x @ k_proj.T # [seq, 2048] # K equals V — halves global KV cache scores = Q @ K.T / sqrt(512) # Full 256K context — no mask out = softmax(scores) @ K

06 — Feed-Forward Network

The MLP: Where Knowledge Lives

After attention, each token passes through an MLP. MicroGPT used a 2-layer ReLU network. Gemma 4 uses a gated MLP — three weight matrices with a learned gate filtering which features pass through.

GeGLU data flow — animated
# Three weights per layer: gate = gelu(x @ gate_proj.T) # gate up = x @ up_proj.T # content out = (gate * up) @ down_proj.T # gate_proj: [21504, 5376] = 115M # up_proj: [21504, 5376] = 115M # down_proj: [5376, 21504] = 115M # Per layer: 346M × 60 = ~20.8B total

Why a Gate?

The gate is a learned filter — gate_proj produces near-zero values for irrelevant features, suppressing them when multiplied with up_proj output. Gated MLPs have better factual recall than plain 2-layer networks.

20.8B Parameters in MLP Alone

346M per layer × 60 layers = 20.8B of the 30.7B total. The MLP is where most of the model's world knowledge is thought to be stored — each of the 21,504 intermediate neurons detects one pattern.

07 — Positional Encoding

RoPE: Position Encoded as Rotation

In zero2ai you learned position encoding as "adding a number to each embedding." MicroGPT used a learned table wpe[16×16]. Gemma 4 uses RoPE — it rotates the Q and K vectors inside each attention head instead of adding anything to the embeddings.

RoPE — each column is a dimension pair, each row is a token position

Each needle shows the rotation angle for that (position, dim-pair). Left = fast rotation (short-range signal). Right = slow (long-range signal). Notice how higher token positions rotate more — that's how the model knows the distance between two tokens.

Two RoPE configs in the same model. Sliding layers: standard RoPE, θ=10,000, all dims rotated — works up to ~8K. Global layers: p-RoPE, θ=1,000,000, only 25% of dims rotated. At 256K token distances standard RoPE wraps around and loses positional signal. p-RoPE prevents this.

# Sliding layers: rope_theta = 10_000.0 # standard rope_type = "default" # all 256 dims rotated # Global layers (p-RoPE): rope_theta = 1_000_000.0 # 100× larger → much slower rotation partial_rotary_factor = 0.25 # only 64 of 256 dims rotated
08 — Vision Encoder

A Transformer Inside a Transformer

For images, Gemma 4 runs a separate 27-layer Vision Transformer on the pixels. Its output gets projected to 5376 dims and concatenated into the text token stream. The decoder never distinguishes image tokens from text tokens.

Image → patches → visual tokens → decoder stream

Why Bidirectional?

Text is causal — each token looks only backward. Images have no order. Patch 47 needs to attend to both patch 12 and patch 89. So the vision encoder uses full bidirectional attention in both directions.

2D RoPE for Images

Text position is 1D (sequence order). Image patch position is 2D (row, column). Standard RoPE would treat patch (2,4) and (3,3) as equal distance even though they're in different rows. 2D RoPE encodes x and y independently — correct position signal at any aspect ratio.

09 — KV Cache

How Generation Stays Fast at 256K Context

In microGPT, every new token required a full forward pass. At 30.7B parameters and 256K context, that's not feasible. The KV Cache saves computed Keys and Values from all previous tokens so only the new token needs processing.

KV cache growing during generation

Naïve KV cache at 256K context (32 heads, 256 dim, 60 layers, K+V) = ~503 GB. Three architectural tricks reduce this to something real hardware can hold.

Trick 1 — GQA
÷ 2
16 KV heads not 32 for sliding layers. 2 query heads share each KV head.
Trick 2 — K = V
÷ 2
Global layers set Keys equal to Values. 4 KV heads, one tensor stored not two.
Trick 3 — Window
÷ 250
Sliding layers only cache the last 1,024 tokens. 50 layers × bounded = O(1024) not O(256K).
10 — Token Generation

From Hidden State to the Next Word

In microGPT: logits = x @ wte.T, then random.choices(). Gemma 4 does the same — plus softcapping before sampling.

Generation simulator
The model processes tokens through 60 decoder layers
Next token logit distribution (after softcap + softmax):
temp=1.0 · top_p=0.95 · top_k=64

Logit Softcapping = 30.0

Before softmax, Gemma 4 applies: logits = tanh(logits / 30.0) × 30.0 — squashes all values into (−30, +30). Without it, a confident model produces one logit of +500 and softmax assigns probability ≈ 1.0 to one token, making generation deterministic. Softcapping preserves diversity.
