00 — What's in the Box

The Repository: 11 Files, 62.6 GB

In microGPT, the entire model fit in ~200 lines of Python. The weights were tiny Python lists. A real model is the same thing — just weights in files. The architecture is still: embeddings → attention → MLP → output.

When you download google/gemma-4-31B-it, you get exactly 11 files. Here's what each one actually does:

Repository File Map — click any file

Click a file above to learn what it does.

The two .safetensors files (49.8 GB + 12.8 GB) are just the weights. "Safetensors" is a security improvement over older pickle-based formats — it prevents malicious code execution when loading weights. Every number in them is a bfloat16: 2 bytes, same range as float32 but lower precision.

01 — Tokenisation

262,144 Tokens: Why So Many?

In microGPT, Karpathy used 27 tokens — one per letter, plus a start/end token. It worked for names. But to speak 140 languages, you need more vocabulary. Gemma 4 uses Byte-Pair Encoding (BPE) — the same algorithm, scaled up.

BPE starts with individual characters, then merges the most common adjacent pairs over and over until it reaches the target vocab size. Common English words become a single token. Rare words break into pieces. Non-English scripts get their own piece sets.

02 — Embeddings & the "Vector DB" Inside

The Embedding Table IS the Vector Database

In zero2ai, you learned that each token gets "a rich profile card" — a vector of numbers representing its meaning. In microGPT, that was wte[27×16] — 27 tokens, each with 16 numbers. Gemma 4 is the exact same thing, just 262,144 tokens × 5,376 numbers each.

When you ask "is there a vector database?" — the embedding table IS it. The 262K × 5376 matrix (that's ~1.4 billion numbers, ~2.8 GB) is where all token "meanings" live, pre-learned during training. Each row is a 5,376-dimensional vector for one token. Similar meanings end up geometrically close. You don't need a separate vector DB — it's baked into the model weights.

The Embedding Lookup — what actually happens

Token ID → row index → 5376-dim vector

embed_tokens: The Main Table

# Shape: [262144, 5376] # Size: 262144 × 5376 × 2 bytes = 2.8 GB # BF16 precision token_id = 4821 # e.g. "▁model" embedding = embed_tokens[4821] # → float vector of shape [5376] # → this IS the token's meaning # Tied to output: lm_head uses the # SAME matrix transposed for output # → saves 2.8 GB of storage

Why 5376 Dimensions?

MicroGPT used 16 dimensions. GPT-2 small uses 768. GPT-3 uses 12,288. More dimensions = more "directions" to encode meaning:

microGPT16~4K
GPT-2 small768117M
GPT-312,288175B

5376 isn't round — it's 3 × 2048 × 7/8. Chosen for divisibility by attention heads while minimising GPU memory fragmentation.

Weight Tying: One Table Does Two Jobs

Gemma 4 sets tie_word_embeddings: true. This means the same 262K × 5376 matrix is used both at input (to convert token IDs to vectors) and at output (to convert the model's final hidden state back to a prediction over 262K tokens). It saves ~2.8 GB and also improves quality — the model learns a single consistent geometry of meaning.

03 — 60 Decoder Layers

Why 60 Layers? And What's the Pattern?

MicroGPT used 2 transformer blocks. Each block: attention → MLP, with layer norm before each. Gemma 4 is the same structure, 60 times. The key difference: not all 60 layers are identical. They alternate between two types of attention.

The pattern from config.json is exact: 5 sliding-window layers, then 1 full-attention layer, repeated 10 times. Every 6th layer (indices 5, 11, 17, 23...) gets to see the whole sequence. The last layer (#59) is always full-attention.

Why this 5:1 ratio? Full attention is O(n²) in sequence length — at 256K tokens, that's 65 billion attention operations per layer. Having 50 sliding-window layers (which are O(n × window) = much cheaper) and only 10 full layers means the model can handle 256K context without running out of memory on a single GPU.

What's Inside Every Layer?

Each of the 60 layers contains exactly 14 named weight tensors stored in the safetensors files. Here's the full anatomy:

04 — Attention: Two Kinds

Sliding Window vs Full Attention (and Why Both)

In zero2ai, you saw attention as "every word looking at every other word". In microGPT, you saw the actual Q·K·V math. Gemma 4 uses the same Q·K·V mechanism — but limits how far back tokens can look in 50 out of 60 layers.

Attention Pattern Visualiser — toggle between modes

Each token attends only within its 1024-token window

The Q·K·V Arithmetic — with Gemma 4's actual numbers

In microGPT, Karpathy used 4 attention heads, head_dim=4. Each head independently asks a question about context. In Gemma 4, sliding layers have 32 query heads but only 16 KV heads — this is Grouped Query Attention (GQA).

# Sliding layer attention math: Q = x @ q_proj.T # [seq, 8192] = 32×256 K = x @ k_proj.T # [seq, 4096] = 16×256 V = x @ v_proj.T # [seq, 4096] = 16×256 # GQA: 2 Q heads share each K,V head # Attention scores: scores = Q @ K.T / sqrt(256) # Apply q_norm and k_norm first # Then mask: only last 1024 positions attn = softmax(scores) # [seq, seq] out = attn @ V # [seq, 4096] out = out @ o_proj.T # [seq, 5376]

New in Gemma 4: Q-Norm and K-Norm

MicroGPT didn't have this. When context is 256K tokens long, the Q·K dot products can become enormous — causing attention to collapse onto a few tokens or spread completely flat. Gemma 4 adds a learned RMSNorm to both Q and K vectors before computing scores:

# After projecting Q and K: Q = rms_norm(Q, q_norm_weight) K = rms_norm(K, k_norm_weight) # q_norm_weight shape: [256] # One per head dimension # This keeps attention stable # at 256K token distances

layer_scalar is another Gemma 4 addition: a single learned number [1] multiplied with each layer's output before the residual add. It's a per-layer "volume dial" — the model learns whether each layer should contribute a lot or a little to the final representation.

Global Layers: K = V (Unified Keys and Values)

In standard attention, K and V are always separate projections. Global layers in Gemma 4 do something unusual: Keys are set equal to Values. This halves the KV cache for global layers — instead of storing 2 separate tensors per layer, you store 1.

05 — Feed-Forward Network (MLP)

The MLP: Where Knowledge Actually Lives

In microGPT, after attention there was a simple two-layer MLP with ReLU. You saw how this is what lets the network model non-linear patterns. Gemma 4 uses a gated MLP — three weight matrices instead of two, with a learned gate that controls which features pass through.

GeGLU: How Gemma 4's MLP Works

# Three weight matrices per layer: gate = gelu(x @ gate_proj.T) # gate signal up = x @ up_proj.T # content out = (gate * up) @ down_proj.T # gate_proj: [21504, 5376] # up_proj: [21504, 5376] # down_proj: [5376, 21504] # Each: ~115M params → 346M/layer # × 60 layers = ~20.8B params total

Why Gated MLP?

The gate acts as a learned filter. gate_proj learns "which concepts are relevant right now?" and produces near-zero values for irrelevant features. Those get multiplied with up_proj's output — suppressing irrelevant content. This is why transformers with gated MLPs have better recall of stored facts.

21,504 = 4 × 5,376

The intermediate size is exactly 4× the hidden size. This "expansion factor" gives the MLP enough room to represent complex transformations. Each of the 21,504 intermediate neurons can detect one specific pattern. With 346M MLP params per layer and 60 layers, the MLP alone accounts for ~20.8B of the 30.7B total. That's where factual knowledge is thought to be stored.

06 — Positional Encoding

RoPE: Teaching the Model Where Words Are

In zero2ai, you learned about position encoding — adding a number to each token embedding to tell the model "this token is in position 4." In microGPT, Karpathy used a simple learned position table (wpe[16×16]). Gemma 4 uses Rotary Position Embeddings (RoPE) — a smarter approach that doesn't add to embeddings but instead rotates the Q and K vectors.

RoPE: Position Encoded as Rotation

Each pair of dims rotates at a different speed

Here's the key intuition: each 256-dim head vector is split into 128 pairs. Each pair is rotated by an angle that depends on the token's position. Early pairs rotate fast (good at detecting close neighbours). Late pairs rotate slowly (good at detecting long-range relationships). When you compute Q·K, the dot product naturally encodes the distance between the two tokens.

Gemma 4 uses two different RoPE configurations: Sliding layers use standard RoPE with θ=10,000 — calibrated for windows up to ~8K tokens. Global layers use Proportional RoPE (p-RoPE) with θ=1,000,000 and partial_rotary_factor=0.25 — meaning only 25% of the head's dimensions get rotated. At 256K token distances, standard RoPE would "wrap around" and lose positional information. p-RoPE prevents this by spreading rotations much more sparsely across the extreme range.

# Sliding layers: rope_theta = 10_000.0 # standard rope_type = "default" # all 256 dims rotated # Global layers (p-RoPE): rope_theta = 1_000_000.0 # 100× larger → slower rotation rope_type = "proportional" partial_rotary_factor = 0.25 # only 64/256 dims rotated # → stable position signal up to 256K tokens
07 — Vision Encoder

Processing Images: A Transformer Inside a Transformer

Everything you've learned applies to text. For images, Gemma 4 runs a separate, smaller transformer (27 layers, 1152 dims) to convert pixels into vectors — then those vectors get projected to match the text decoder's 5376 dims and concatenated into the token stream. The decoder never "knows" it's looking at image tokens vs text tokens.

Why Bidirectional Attention?

Text is causal — each token can only look at past tokens. Images have no order. Patch 47 should be able to attend to patch 12 and patch 89. So the vision encoder uses full bidirectional attention — every patch sees every other patch in both directions.

2D RoPE for Patches

In text, position is 1D (token 1, token 2...). For image patches, position is 2D (row 2, col 4). Standard RoPE would treat patch (2,4) and patch (3,3) as nearby in sequence, even though they're in different rows. 2D RoPE encodes both x and y independently — a patch in column 4 always gets the same x-rotation regardless of which row it's in.

This is how Gemma 4 handles variable aspect ratios — a 16:9 landscape image and a 9:16 portrait image both get correct 2D positions without distortion.

08 — The KV Cache

How Generation Stays Fast: Caching Computed Keys and Values

In microGPT, every time you generate a new token, the whole forward pass reruns from scratch. At 200 lines with 27 vocab tokens this is fine. At 256K context with 30.7B parameters, you cannot recompute everything for every new token. The KV Cache saves the Key and Value vectors from every previous token so only the new token needs to be processed.

KV Cache Growing During Generation

Generating tokens...

At 256K context length with a naïve approach (32 heads, 256 dim, 60 layers, 2 tensors), the KV cache would be 256K × 32 × 256 × 2 × 60 × 2 bytes = 503 GB. Gemma 4 uses three tricks to bring this down dramatically:

Trick 1: GQA
÷2
Grouped Query Attention: 16 KV heads instead of 32 for sliding layers. 2 query heads share each KV head. Halves the sliding KV cache.
Trick 2: K=V
÷2
In global layers, Keys are set equal to Values. Only 4 KV heads, with a single K/V tensor. Near-zero quality loss.
Trick 3: Window Bound
÷250
Sliding layers only need to cache the last 1,024 tokens, not all 256K. 50 layers × bounded window = O(1024) not O(256K).
09 — Normalisation

RMSNorm: Four Per Layer, and Why

In microGPT, Karpathy used LayerNorm — the standard normalisation that subtracts the mean and divides by the standard deviation. Gemma 4 uses RMSNorm (Root Mean Square Norm) which skips subtracting the mean. It's faster and works just as well for large models.

Each layer has four normalisation steps, each with its own learned weight vector of shape [5376]:

# RMSNorm formula: def rms_norm(x, weight, eps=1e-6): rms = sqrt(mean(x**2) + eps) return (x / rms) * weight # weight shape: [5376] — one scale per dimension # eps=1e-6 prevents division by zero # 4 normalisation steps per layer: input_layernorm # before attention post_attention_layernorm # after attention, before residual add pre_feedforward_layernorm # before MLP post_feedforward_layernorm # after MLP, before residual add

Normalisation keeps the numbers in a reasonable range as they flow through 60 layers. Without it, values would explode or vanish — gradient descent couldn't work.

10 — Generating Output

From Hidden State to Next Token

In microGPT, after the transformer blocks you saw a simple matrix multiply: logits = x @ wte.T, then random.choices to sample. Gemma 4 does the same thing — with two additions: logit softcapping and the thinking mode output format.

Token Generation Simulator

The model processes tokens through 60 decoder layers
Next token logits (after softcap + softmax):
temp=1.0 · top_p=0.95 · top_k=64

final_logit_softcapping = 30.0

Before softmax, Gemma 4 applies: logits = tanh(logits / 30.0) × 30.0. This squashes all logit values into the range (−30, +30). Without it, a highly confident model might produce one logit of +500 — softmax would then assign probability ≈ 1.0 to that token, making generation completely deterministic. Softcapping prevents this collapse.

Temperature, top_p, top_k — these three hyperparameters are set in generation_config.json. Temperature scales the logits before softmax (lower = sharper = more confident, higher = flatter = more random). Top_k keeps only the 64 most likely tokens. Top_p then keeps the smallest set whose cumulative probability exceeds 0.95. Tokens outside that set get probability 0.

Thinking Mode: a Channel Inside the Output

11 — Putting It All Together

One Forward Pass, Start to Finish

Here's every step that happens when Gemma 4 processes one token — from raw text to predicted next token. This is the complete picture connecting everything above:

Complete Forward Pass — animated step by step

  1. Tokenise — raw text split into BPE tokens, mapped to integer IDs
  2. Embed — each ID looked up in the 262K × 5376 embedding table → a vector per token
  3. Vision merge (if images) — ViT encoder processes patches, embed_vision projects them to 5376 dims, concatenated into token stream
  4. 60× Decoder layer — for each layer: RMSNorm → attention (sliding or global) → residual → RMSNorm → MLP (GeGLU) → residual × layer_scalar
  5. Final RMSNorm — one last normalisation on the final hidden state
  6. LM Head — multiply by embedding table transposed → 262K raw logits
  7. Softcap — tanh(logits/30) × 30
  8. Sample — apply temperature, top_k=64, top_p=0.95, pick a token
  9. Repeat — new token appended, process repeats (KV cache means only new token needs full compute)
12 — Results

What All This Architecture Achieves

Every architectural choice above — hybrid attention, GQA, K=V unification, p-RoPE, q/k-norm, layer_scalar, GeGLU — compounds into benchmark improvements that are, frankly, shocking compared to Gemma 3.

Why such a large jump from Gemma 3? The 256K context window (vs 128K) enabled reasoning over longer chains. The hybrid attention architecture made that context efficient. The thinking mode added explicit chain-of-thought reasoning. And the training mix included harder math and coding problems. All those compounded — AIME 2026 went from 20.8% to 89.2%, nearly 4× improvement on math olympiad problems.
